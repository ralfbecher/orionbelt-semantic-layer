    OrionBelt v1.0 Gap Analysis vs. Malloy, LookML, dbt SL, GoodData, BSL¶

    Date: 2026-03-18 Purpose: Identify feature gaps before promoting to version 1.x Competitors analyzed: Malloy, LookML (Looker), dbt Semantic Layer (MetricFlow), GoodData, Boring Semantic Layer (BSL)

    TIER 1 — Must-Have (Table Stakes for v1.0)¶

    1. Descriptions / Labels on All Objects¶

    Who has it: Everyone (LookML description:, dbt description+label, GoodData metadata, Malloy self-documenting, BSL description on models/dimensions/measures)

    OrionBelt status: You have owner, synonyms, format — but no description field on dimensions, measures, metrics, or data objects.

    Gap: Without descriptions, no BI tool or AI agent can present meaningful context to users.

    2. Filtered Measures (Measure-Level WHERE)¶

    Who has it: Malloy (count() { where: state = 'CA' }), LookML (filters: on measures), dbt (metric-level filter:), GoodData (SELECT SUM(x) WHERE attr IN (...)), BSL (base filter: on models + lambda filters in queries)

    OrionBelt status: You have MeasureFilter — but only a single filter per measure with one column reference and basic operators.

    Gap: Users need to define things like "US Revenue" = SUM(revenue) WHERE country = 'US' without creating separate measures. Need multiple filters per measure with AND/OR logic.

    3. Cumulative / Running Metrics¶

    Who has it: dbt (cumulative metric type with window and grain_to_date), GoodData (RUNSUM, RUNAVG, RUNMIN, RUNMAX with rolling windows), LookML (running_total measure type), BSL (partial — via .mutate() with xorq window functions, not declarative)

    OrionBelt status: Nothing. No running totals, no trailing windows, no grain-to-date resets.

    Gap: MTD/YTD revenue, 7-day rolling averages, and running totals are among the most common analytical patterns.

    4. Period-over-Period / Time-Shifting¶

    Who has it: GoodData (FOR Previous, FOR PreviousPeriod, FOR Next), dbt (derived metrics with offset_window / offset_to_grain), LookML (period_over_period measure type + table calcs). BSL also lacks this.

    OrionBelt status: Nothing. You have time grains but no time-shifting capability.

    Gap: "Revenue vs same period last year", "MoM growth", "QoQ change" are bread-and-butter analytics that every user expects.

    5. Ratio Metrics¶

    Who has it: dbt (ratio metric type with numerator/denominator), GoodData (metric division with automatic grain handling), Malloy (measure arithmetic), BSL (calculated measures: _.revenue / _.order_count)

    OrionBelt status: You have expression-based metrics ({[Revenue]} / {[Quantity]}), which is close — but lacks automatic grain alignment when numerator and denominator come from different fact tables, and lacks separate filters per component.

    Gap: Ratio metrics with per-component filters (e.g., "conversion rate" = filtered_conversions / filtered_visits) are a core pattern.

    6. Derived Metrics (Metric-on-Metric Composition)¶

    Who has it: dbt (derived metrics referencing other metrics via input_metrics), GoodData (metrics referencing metrics: SELECT {metric/a} - {metric/b}), LookML (type: number measures referencing other measures), BSL (calculated measures reference other measures, but no separate metric layer)

    OrionBelt status: Metrics reference measures only ({[Measure Name]}). You cannot compose metrics from other metrics.

    Gap: Building a metric tree (e.g., Profit = Revenue - Cost, then Profit Margin = Profit / Revenue) requires metric-on-metric references.

    7. Non-Additive / Semi-Additive Measures¶

    Who has it: dbt (non_additive_dimension — use MIN/MAX over time instead of SUM for snapshot measures like account balances), GoodData (additive vs non-additive facts), BSL (automatic MEAN decomposition to SUM/COUNT in pre-aggregation for correctness)

    OrionBelt status: You have NumClass enum (categorical, additive, non-additive) on columns, but it's only used for validation — not enforced in query planning.

    Gap: Account balances, inventory levels, headcount snapshots all break if summed over time. The planner needs to auto-apply MAX/MIN over the time dimension for non-additive measures.

    TIER 2 — Strongly Expected (Competitive Differentiators)¶

    8. Level-of-Detail (LOD) / Ungrouped Aggregates¶

    Who has it: Malloy (all(), exclude() — percent-of-total, sub-group totals), GoodData (BY, BY ALL, BY ALL IN ALL OTHER DIMENSIONS), LookML (table calculations with :total), BSL (t.all() for ungrouped aggregates — t.distance.sum() / t.all(t.distance.sum()) * 100)

    OrionBelt status: You have total: true for grand totals via AGG(x) OVER (). But no arbitrary LOD expressions — no "percent of parent", no "exclude dimension X from grouping".

    Gap: Percent-of-total, contribution analysis, and benchmark comparisons all need LOD control.

    9. Model Composition / Multi-File / Imports¶

    Who has it: Malloy (import "file.malloy"), LookML (include:, extends, refinements, cross-project refs), dbt (packages, dbt Mesh cross-project refs)

    OrionBelt status: Single YAML file per model. No imports, no extends, no cross-model references.

    Gap: Real-world models get large. Teams need to split models across files, share base definitions, and layer customizations.

    10. Row-Level Security (RLS)¶

    Who has it: LookML (access_filter + user attributes), GoodData (User Data Filters, Workspace Data Filters), dbt (service tokens + model access)

    OrionBelt status: Nothing. No user context, no row filtering.

    Gap: Any multi-tenant or governed deployment needs RLS. Without it, OrionBelt can't serve enterprise workloads.

    11. Parameterization / Dynamic Filters¶

    Who has it: LookML (parameters, templated filters, Liquid variables), GoodData (filtered variables, numerical variables), Malloy (experimental parameterized sources), dbt (Jinja filter templates)

    OrionBelt status: Nothing. Queries are static; no user-scoped parameters or dynamic SQL injection.

    Gap: Dashboard filter widgets, user-specific defaults, and dynamic SQL generation all depend on parameterization.

    12. Caching / Pre-Aggregation / Saved Queries¶

    Who has it: LookML (PDTs, incremental PDTs, aggregate awareness, datagroups), dbt (saved queries, exports, declarative caching), GoodData (FlexCache, Apache Arrow + Iceberg), BSL (xorq auto-caching at execution layer)

    OrionBelt status: Nothing. Every query hits the database.

    Gap: At scale, query latency and cost become problems. Even basic result caching or saved-query materialization would help.

    13. Window Functions Beyond Totals¶

    Who has it: Malloy (all(), exclude()), GoodData (RANK, DENSE_RANK, RUNSUM with window boundaries), LookML (running_total, percent_of_total, percent_of_previous), BSL (.mutate() with xorq windowing — dense_rank(), rolling_avg(), running_total())

    OrionBelt status: WindowFunction AST node exists, used only for total: true grand totals. No RANK, ROW_NUMBER, LAG, LEAD, NTILE.

    Gap: Ranking, moving averages, and lag/lead are standard analytical operations.

    14. Data Tests / Assertions¶

    Who has it: LookML (test: with assert:), dbt (model tests + dbt sl validate), GoodData (LDM + MAQL validation)

    OrionBelt status: Semantic validation (model structure) and SQL validation (post-gen sqlglot). No data-level tests (e.g., "revenue is never negative", "every order has a customer").

    Gap: Governance requires provable data quality guarantees.

    15. Fiscal / Custom Calendars¶

    Who has it: LookML (fiscal_month_offset, custom calendar tables), GoodData (custom date dimensions), dbt (custom time spine with fiscal granularities)

    OrionBelt status: Nothing. Only standard calendar grains (year/quarter/month/week/day/hour/minute/second).

    Gap: Retail (4-4-5), fiscal year offsets, and ISO week handling are common requirements.

    Priority Recommendation for v1.0¶

    Based on what every competitor has and users will immediately notice missing:

    MUST SHIP (Tier 1)¶

    1. description field on all model objects
    2. Multi-filter measures (AND/OR logic)
    3. Cumulative metrics (running totals, grain-to-date)
    4. Period-over-period (time-shift functions)
    5. Ratio metrics with per-component filters
    6. Metric-on-metric composition
    7. Semi-additive measure enforcement

    SHIP SOON AFTER (Tier 2)¶

    1. LOD expressions (all/exclude)
    2. Multi-file model imports
    3. Row-level security
    4. Parameterization
    5. Caching / saved queries
    6. RANK/LAG/LEAD window functions
    7. Data tests
    8. Fiscal calendars

    Note: Items 1–4 are the most glaring — every single competitor has them, and their absence would immediately signal "not production-ready" to evaluators.

    Detailed Competitor Feature Matrix¶

    Malloy — Key Features¶

    Category Features
    Modeling Sources, dimensions, measures, views, primary keys, joins with explicit cardinality
    Queries group_by + aggregate, select (projection), nest (hierarchical), pipelining (->), refinement (+)
    Joins join_one:, join_many:, join_cross: — cardinality-driven symmetric aggregates, fan-out protection
    Expressions Arithmetic, comparisons, boolean, null coalescing (??), regex, pick/when/else (CASE), raw SQL escape (sql_number() etc.)
    Types number, string, boolean, date, timestamp, json, record, array — type inference, aggregate/scalar distinction
    Composition Source extension (extend), query refinement (+), imports, layered modeling
    Time Time literals (@2023), truncation (.year, .month), extraction, ranges, date arithmetic, timezone support
    LOD all() (grand total), exclude() (remove dimensions), aggregate locality (source.count())
    Dialects DuckDB, BigQuery, Snowflake, PostgreSQL, MySQL, Trino, Presto
    Integration VS Code extension, CLI, Publisher (REST + MCP), TypeScript SDK, Python SDK, MalloySQL
    Missing No RLS, no caching, no data tests, no fiscal calendars

    LookML — Key Features¶

    Category Features
    Modeling Explores, views, dimensions, measures, dimension groups, derived tables (SQL + NDT), sets
    Measures 16+ types: sum, count, count_distinct, average, min, max, sum_distinct, average_distinct, list, median, percentile, number, percent_of_total, percent_of_previous, running_total, period_over_period
    Joins left_outer, inner, full_outer, cross — relationship declarations (many_to_one, etc.) — symmetric aggregates via primary key hashing
    Expressions SQL blocks, Liquid templating, type: yesno, type: tier, type: case
    Composition extends, refinements, includes, cross-project refs via manifest, constants
    Time dimension_group (auto 20+ timeframes), duration groups, convert_tz, fiscal_month_offset, custom calendars
    Security access_grants, required_access_grants (field-level), access_filter (row-level), user attributes
    Caching Datagroups, persist_with, PDTs, incremental PDTs, aggregate tables with auto-routing
    Parameterization Parameters, templated filters, Liquid variables, user attributes
    Testing LookML validator, content validator, data tests (test: + assert:), SQL validator, CI/CD (Spectacles)
    Dialects 50+ dialects (BigQuery, Snowflake, Redshift, Databricks, Postgres, MySQL, SQL Server, ClickHouse, etc.)
    Integration REST API, SDKs (Python, Ruby, TS, Kotlin, Swift, Go), Open SQL Interface (JDBC), embedded analytics, actions

    dbt Semantic Layer (MetricFlow) — Key Features¶

    Category Features
    Modeling Semantic models (1:1 with dbt models), entities (primary/foreign/unique/natural), dimensions (categorical/time/SCD2), measures
    Metrics 5 types: simple, derived (arbitrary expressions), ratio (numerator/denominator), cumulative (window/grain_to_date), conversion (funnel)
    Joins Entity-based implicit joins — no explicit JOIN ON — fan-out prevention via entity types, multi-hop (up to 2)
    Time metric_time, time spine, custom granularities, cumulative windows, period-over-period (offset_window/offset_to_grain), fill_nulls_with, join_to_timespine, non-additive dimensions
    Composition Metrics reference other metrics, create_metric on measures, dbt packages, dbt Mesh cross-project refs, saved queries
    Security Service tokens (scoped), dbt Cloud RBAC, model access modifiers (public/protected/private), private metrics
    Caching Saved queries, exports (table/view materialization), declarative caching
    Testing 3-level validation (parsing, semantic, data platform), CI/CD integration, dbt tests on source models
    Dialects Snowflake, BigQuery, Databricks, Redshift, PostgreSQL, Trino, DuckDB
    Integration JDBC (Arrow Flight SQL), ADBC, GraphQL API, Python SDK, Tableau, Power BI, Google Sheets, Hex, Mode, MCP server

    GoodData — Key Features¶

    Category Features
    Modeling Datasets, facts (additive/non-additive), attributes, labels (text/hyperlink/image/geo), date dimensions, references
    Query Language MAQL: SELECT, WHERE, BY/BY ALL, WITHOUT PARENT FILTER, FOR Previous/Next/PreviousPeriod/NextPeriod, IF/THEN/ELSE, CASE/WHEN
    Joins Reference-based (implicit), star/snowflake schema, many-to-many via bridge datasets
    Time Built-in date dimensions (1900-2050), relative date filters, YTD/QTD/MTD/WTD, FOR Previous/Next, RUNSUM/RUNAVG with rolling windows, forecasting
    LOD BY (lock granularity), BY ALL (ignore dimension), BY ALL IN ALL OTHER DIMENSIONS (grand total), BY ALL EXCEPT (keep one dimension)
    Composition Named reusable metrics, nested metrics (metric references metrics), workspace inheritance
    Security Workspace permissions, User Data Filters (RLS), Workspace Data Filters, dashboard permissions, filtered/numerical variables
    Caching FlexCache (Apache Arrow in-memory + Iceberg durable), manual/auto invalidation
    Advanced RANK/DENSE_RANK, TOP(n)/BOTTOM(n), running totals with window boundaries, forecasting (ML), clustering, anomaly detection
    Multi-tenancy Workspace hierarchy, data isolation, template-based provisioning, per-workspace data sources
    Dialects 20+ (Snowflake, BigQuery, Databricks, Redshift, PostgreSQL, MySQL, SQL Server, ClickHouse, Dremio, etc.)
    Integration REST API, declarative API (YAML/JSON), React SDK, Python SDK, gooddata-pandas, FlexConnect (Flight RPC), VS Code extension

    Boring Semantic Layer (BSL) — Key Features¶

    Category Features
    Modeling SemanticTable wrapping Ibis table expressions; dimensions (regular, entity, time), measures, calculated measures, joins with cardinality
    Definition Python-first (fluent API + lambdas) OR YAML; Ibis deferred expressions (_.column) as expression language
    Queries Method chaining (.filter().group_by().aggregate().mutate().order_by().limit()) + parameter-based query() API
    Joins join_one / join_many / join_cross — cardinality-aware with automatic pre-aggregation to prevent fan traps and chasm traps
    Fan-Out Pre-aggregation at native grain before joining; MEAN auto-decomposed to SUM/COUNT; COUNT DISTINCT deferred — unique among competitors
    Expressions Ibis expressions (lambdas, deferred _), AST-validated string expressions in YAML, calculated measure AST (MeasureRef, BinOp)
    LOD t.all() for ungrouped aggregates (percent-of-total: t.x.sum() / t.all(t.x.sum()) * 100)
    Time Time dimensions with smallest_time_grain validation, time_grain= and time_range= query params; no period-over-period or cumulative
    Composition Runtime model enhancement (.with_measures()), multi-model YAML, join-based composition; no formal imports/extends
    Nested Data Automatic unnesting of struct/array fields (experimental, Malloy-inspired)
    Window Functions Via .mutate() + xorq windowing: dense_rank(), rolling averages, running totals — imperative, not declarative
    Visualization Built-in: Altair, Plotly, ECharts, Plotext; dashboard rendering with 16-column CSS grid
    AI Integration MCP server (4 tools: list_models, get_model, query_model, search_dimension_values), LangGraph/LangChain agent, OpenAI tools
    Dialects 20+ via Ibis backends (DuckDB, Snowflake, BigQuery, PostgreSQL, ClickHouse, Databricks, Trino, MySQL, Oracle, MSSQL, etc.)
    Caching Via xorq auto-caching at execution layer; projection pushdown always on
    Lineage Dependency graph (BFS traversal, JSON export, predecessors/successors)
    Output pandas, PyArrow, Polars, CSV, Parquet, JSON, XLSX, compiled SQL
    Missing No REST API (library only), no RLS, no period-over-period, no fiscal calendars, no data tests, no parameterization

    Head-to-Head: OrionBelt vs. BSL¶

    BSL is the closest competitor in philosophy (lightweight, open-source, Python) but takes a fundamentally different architectural approach:

    Aspect OrionBelt BSL
    Architecture Custom SQL AST + codegen per dialect Ibis expression graph → backend compiles SQL
    Model Format YAML-only (OBML) Python API + YAML
    API Full REST API (30+ endpoints) + MCP (separate repo) Python library only + built-in MCP
    Fan-Out Strategy FanoutError detection + CFL planner (UNION ALL) Pre-aggregation at native grain before join
    Multi-Fact CFL with UNION ALL + NULL padding Chasm trap handling via multi-arm pre-aggregation
    Dialects 7 (custom codegen per dialect) 20+ (inherited from Ibis, zero dialect code)
    LOD / Ungrouped total: true (grand total only) t.all() (percent-of-total)
    Window Functions Grand totals only Full via .mutate() (rank, rolling, running)
    Visualization None (Gradio UI for queries) 4 charting backends + dashboard grid
    Expression Safety SQL template parsing AST-validated safe_eval() sandbox
    Nested Data Not supported Automatic struct/array unnesting
    Session Management TTL-scoped sessions with model registry Stateless library

    Where OrionBelt leads: REST API surface, session management, SQL explain/lineage, OBML as a portable declarative format, secondary join paths, dimensionsExclude anti-joins, OSI interoperability.

    Where BSL leads: Dialect breadth (20+ vs 7), LOD/window functions, pre-aggregation correctness, built-in visualization, Python-first ergonomics, AI agent integrations (LangGraph + OpenAI + MCP), nested data support.
